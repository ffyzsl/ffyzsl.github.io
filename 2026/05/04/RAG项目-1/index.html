RAG 菜谱检索与生成系统

发表于 2026-05-04

RAG 菜谱检索与生成系统 — 技术文档

一、项目概述

本项目是一个基于 RAG（Retrieval-Augmented Generation，检索增强生成）架构的智能菜谱问答系统。简单来说，这个系统可以理解用户用自然语言提出的问题（比如”红烧肉怎么做？”或者”推荐几个简单的素菜”），然后从本地存储的 323 个中文菜谱中检索出最相关的内容，最后调用大语言模型生成一个结构清晰、实用详细的回答。github链接：https://github.com/ffyzsl/cook-rag-1

整个系统采用经典的 RAG 四模块架构，由 main.py 中的 RecipeRAGSystem 类统一调度。四个核心模块分别是：

  1. 数据准备模块（Data Preparation）：负责从 Markdown 文件中加载菜谱，并提取其中的结构化信息（分类、难度、菜名等），然后将长文档切分成便于检索的小块。
  2. 索引构建模块（Index Construction）：负责将切分好的文本块转换成向量（Embedding），并构建 FAISS 向量索引，支持后续的语义搜索。
  3. 检索优化模块（Retrieval Optimization）：负责实现混合检索策略——同时使用基于语义的 FAISS 向量检索和基于关键词的 BM25 稀疏检索，然后通过 RRF 算法融合排序，提升检索质量。
  4. 生成集成模块（Generation Integration）：负责与大语言模型交互，根据检索到的菜谱内容生成最终回答，同时提供查询路由（判断用户意图）和查询重写（优化模糊查询）功能。

技术栈方面，本项目使用 LangChain 作为 RAG 框架，阿里云百炼平台的 DashScope 作为 Embedding 模型，通义千问（Qwen）作为生成式大语言模型，FAISS 作为向量数据库。

三、配置系统（config.py）

3.1 RAGConfig 数据类

config.py 是整个系统的配置中心。它定义了一个名为 RAGConfig 的数据类（使用 Python 的 @dataclass 装饰器），集中管理所有可调参数。

3.2 设计要点

  • from_dict 类方法：允许从字典（比如从 JSON 配置文件读取的内容）直接创建 RAGConfig 实例。这在需要从外部文件加载配置时非常有用。
  • to_dict 方法：将配置对象序列化回字典，方便保存或打印。
  • DEFAULT_CONFIG 全局实例：使用默认参数创建的单例配置对象，其他模块直接 from config import DEFAULT_CONFIG 即可使用，避免了在多个地方重复写配置代码。

使用 @dataclass 装饰器的好处是 Python 会自动生成 __init__、__repr__、__eq__ 等方法，代码非常简洁。__post_init__ 方法目前为空，是一个预留的扩展点，可以在初始化完成后做一些额外的校验或处理。

四、数据准备模块（rag_modules/data_preparation.py）

4.1 模块职责

数据准备模块是整个 RAG 系统的第一步，它负责将散落在磁盘上的 323 个 Markdown 菜谱文件加载到内存中，从中提取出结构化的元数据信息，然后将每个完整的菜谱文档按照 Markdown 标题层级切分成小块，为后续的向量化和检索做准备。

4.2 数据结构设计

这个模块使用了父子文档模式（Parent-Child Document Pattern），这是 RAG 系统中一个非常重要的设计模式：

  • 父文档（Parent Document）：一个完整的菜谱 Markdown 文件。例如 红烧肉.md 的全部内容就是一个父文档，它包含菜品介绍、食材清单、制作步骤等完整信息。
  • 子文档/子块（Child Chunk）：父文档按 Markdown 标题（#、##、###）切分后的小块。例如”## 食材”下面的内容就是一个子块，”## 步骤”下面的内容是另一个子块。

这样设计的原因在于：

  • 检索时用子块：子块粒度细，语义更聚焦（比如一块专门讲”食材”，另一块专门讲”步骤”），检索时能更精准地匹配用户的问题。
  • 生成回答时用父文档：当某个子块被检索命中后，系统通过 parent_child_map 映射关系找回它所属的完整父文档，这样 LLM 能看到完整的菜谱上下文，生成的回答才不会”断章取义”。

4.3 CATEGORY_MAPPING 分类映射

菜谱文件存放在按分类命名的子目录下，目录名是英文，但展示给用户时需要中文。因此模块定义了一个类变量 CATEGORY_MAPPING：

此外还有 DIFFICULTY_LABELS，定义了五个难度等级：”非常简单”、”简单”、”中等”、”困难”、”非常困难”。

4.4 load_documents() 文档加载

这个方法是数据准备的入口。它的工作流程如下：

  1. 使用 Path(self.data_path).rglob("*.md") 递归遍历数据目录下所有的 .md 文件。rglob 是 pathlib.Path 的方法，不管文件在多深的子目录中都能找到。
  2. 对每个 Markdown 文件：
    • 用 UTF-8 编码读取文件内容。
    • 计算文件相对于数据根目录的路径，然后用 MD5 哈希生成一个 32 位的十六进制字符串作为 parent_id。使用 MD5 的好处是：相同文件每次生成的 ID 完全一致，不同文件几乎不可能产生相同的 ID。
    • 创建一个 LangChain 的 Document 对象，将文件内容存入 page_content，将 source（源文件路径）、parent_id、doc_type="parent" 等基础信息存入 metadata。
  3. 对每个加载的文档调用 _enhance_metadata() 补充元数据。
  4. 将加载的文档列表存入 self.documents。

4.5 _enhance_metadata() 元数据增强

这个方法负责从文档中提取更丰富的结构化信息，这是实现按分类/难度过滤检索的基础。它会从三个维度增强元数据：

提取分类（category）：

  • 将文件的完整路径拆分成各组成部分（通过 Path.parts）。
  • 遍历 CATEGORY_MAPPING，检查路径中是否包含英文目录名（如 meat_dish），找到后将其映射为中文分类（如”荤菜”）。
  • 如果找不到任何匹配，默认分类为”其他”。

提取菜名（dish_name）：

  • 直接取文件名去掉 .md 后缀作为菜名。这是一种约定：文件名就是菜谱名称，例如 红烧肉.md 的菜名就是”红烧肉”。

提取难度（difficulty）：

  • 通过统计文档内容中 ★（实心五角星）的数量来判断难度等级。
  • 判断逻辑是从多到少匹配，避免子串误判。例如先检查 ★★★★★（非常困难），再检查 ★★★★（困难），以此类推。这是因为 ★★★★★ 包含子串 ★，如果先匹配 ★ 就会错误地将”非常困难”判定为”非常简单”。
  • 没有任何星号标记的默认为”未知”。

4.6 chunk_documents() 文档分块

这个方法是实现父子文档模式的关键。它将每个父文档按 Markdown 标题结构切分成子块：

  1. 调用内部方法 _markdown_header_split() 执行实际的分块逻辑。
  2. 为每个子块补充元数据：生成唯一的 chunk_id（使用 uuid.uuid4()），记录 batch_index（在批次中的序号）和 chunk_size（块内容长度）。

4.7 _markdown_header_split() 分块核心逻辑

这是实际执行分块的方法，使用了 LangChain 提供的 MarkdownHeaderTextSplitter：

  1. 定义分割标题级别：指定在遇到 #（主标题）、##（二级标题）、###（三级标题）时切开文档。注意 strip_headers=False 保留了标题文本在分块中，这有助于后续理解块的上下文。
  2. 遍历每个父文档：
    • 先快速检查文档中是否包含任何以 # 开头的行。如果没有，说明该文档缺少 Markdown 标题结构，记录一条警告日志。
    • 调用 markdown_splitter.split_text(doc.page_content) 按标题切分，返回一个子 Document 列表。
    • 对每个子块：生成 chunk_id、继承父文档的元数据（分类、菜名、难度等）、设置 doc_type="child"、记录在父文档中的 chunk_index。
    • 最关键的一步：将 child_id → parent_id 的映射关系存入 self.parent_child_map 字典。这是后续从子块找回完整父文档的桥梁。

4.8 其他辅助方法

  • **filter_documents_by_category(category)**：按分类过滤父文档，使用列表推导式简洁实现。
  • **filter_documents_by_difficulty(difficulty)**：按难度过滤父文档，逻辑同上。
  • **get_statistics()**：汇总统计信息，包括文档总数、分块总数、分类分布、难度分布、平均块大小。这个方法在系统启动时被调用来展示知识库概况。
  • **export_metadata(output_path)**：将文档的关键元数据导出为 JSON 文件，方便离线分析或检查数据质量。
  • **get_supported_categories() 和 get_supported_difficulties()**：类方法，对外提供支持的分类和难度标签列表，供 main.py 中的关键词过滤功能使用。

4.9 get_parent_documents() 父子文档逆映射

这是 RAG 检索流程中的关键环节。它接收一组检索到的子块，根据 parent_child_map 找回对应的完整父文档：

  1. 遍历所有子块，取出每个子块的 parent_id，累加每个父文档被命中的子块数量（作为相关性评分）。
  2. 从 self.documents 中查找并缓存父文档对象，避免重复遍历。
  3. 将父文档按命中次数降序排列——被命中的子块越多，说明该菜谱与查询越相关。
  4. 返回去重后的父文档列表，并通过日志展示每个父文档的菜名和命中块数。

五、索引构建模块（rag_modules/index_construction.py）

5.1 模块职责

索引构建模块负责将文本块的文字内容转换成计算机能理解的向量（数字数组），然后基于这些向量构建 FAISS 索引，使得系统能够通过”语义相似度”快速找到与用户问题最相关的文本块。

5.2 向量化的原理

向量化是 RAG 系统的核心技术之一。可以这样理解：人类的语言是文字，但计算机只认识数字。Embedding 模型就像一个”翻译官”，它能把一句话翻译成一个固定长度的数字数组（比如 768 个数字），这个数组就是”向量”。

神奇的是，这个向量不是随机的。Embedding 模型经过大量语料的训练，学会了把语义相近的文本映射到向量空间中距离相近的位置。比如”红烧肉的做法”和”红烧肉的步骤”这两句话，它们生成的向量在数学上非常接近（比如余弦相似度很高）；而”红烧肉的做法”和”天气预报”这两个毫不相干的话题，它们的向量距离就很远。

5.3 类设计

三个核心属性：

  • model_name：使用的 Embedding 模型名称。实际运行时使用的是 config.py 中配置的 text-embedding-v2（阿里云百炼平台提供的模型），而不是这里默认的 BGE 模型。这是因为 main.py 在创建模块实例时传入了 model_name=self.config.embedding_model 覆盖了默认值。
  • embeddings：LangChain 封装的 DashScope 嵌入对象，对外提供 embed_query()（将单条查询文本转为向量）和 embed_documents()（将多条文档文本转为向量）两个接口。
  • vectorstore：FAISS 向量存储实例，管理所有向量索引，支持相似度搜索。

5.4 setup_embeddings() 初始化嵌入模型

这个方法首先检查环境变量 DASHSCOPE_API_KEY 是否存在，如果不存在直接抛出异常（因为后续所有操作都依赖这个 API Key）。然后创建 DashScopeEmbeddings 实例并赋值给 self.embeddings。

DashScopeEmbeddings 是 LangChain 社区包提供的对阿里云百炼 Embedding API 的封装。它内部处理了 API 调用的细节，我们只需要传入模型名称和 API Key 即可。

5.5 build_vector_index(chunks) 构建向量索引

这是索引构建的核心方法，接收从数据准备模块传来的文档块列表：

  1. 检查 chunks 是否为空，空列表直接抛异常（构建空索引没有意义）。
  2. 调用 FAISS.from_documents(documents=chunks, embedding=self.embeddings)。
    • LangChain 的 FAISS.from_documents() 方法会自动做两件事：
      • 调用 self.embeddings.embed_documents() 将每个文档块的文本转为向量。
      • 将所有向量组织成 FAISS 索引结构，以便后续高效检索。
  3. 将构建好的 FAISS 对象存入 self.vectorstore，并返回它。

5.6 save_index() 保存索引

将内存中的 FAISS 索引持久化到磁盘。调用 self.vectorstore.save_local(self.index_save_path)，FAISS 会在指定目录下生成两个文件：

  • index.faiss：存储向量数据本身。
  • index.pkl：存储与向量关联的文档元数据（使用 pickle 序列化）。

在调用 save_local 之前，使用 Path(self.index_save_path).mkdir(parents=True, exist_ok=True) 确保目标目录存在。

5.7 load_index() 加载索引

从磁盘加载之前保存的 FAISS 索引：

  1. 如果 self.embeddings 为 None（可能没有正确初始化），先调用 setup_embeddings() 补救。
  2. 检查索引目录是否存在，不存在则返回 None（调用方可以根据返回值判断是否需要构建新索引）。
  3. 调用 FAISS.load_local(self.index_save_path, self.embeddings, allow_dangerous_deserialization=True)。
    • 需要传入 self.embeddings，因为 FAISS 在检索时需要将用户的查询文本转换为向量，而这个转换需要用到同一个 Embedding 模型。
    • allow_dangerous_deserialization=True 是安全性参数。因为 index.pkl 文件使用 pickle 反序列化，如果文件来源不可信可能存在安全风险。在本项目中，索引文件是本机生成的，所以可以放心使用。
  4. 如果加载失败（文件损坏、版本不兼容等），捕获异常并返回 None，让系统走重建索引的路径。

5.8 similarity_search(query, k) 相似度检索

将用户的自然语言查询文本向量化，然后在 FAISS 索引中找到最相似的 k 个文档块：

  1. 检查 self.vectorstore 是否为 None。
  2. 调用 self.vectorstore.similarity_search(query, k=k)。FAISS 内部会先用 self.embeddings.embed_query(query) 将查询转为向量，然后使用高效的近似最近邻算法找到相似度最高的 k 个向量，返回对应的 Document 列表。

5.9 add_documents(new_chunks) 增量添加

支持在已有索引的基础上追加新文档块，而不需要重建整个索引。调用 self.vectorstore.add_documents(new_chunks) 即可。

六、检索优化模块（rag_modules/retrieval_optimization.py）

6.1 模块职责

检索优化模块实现了混合检索（Hybrid Search）策略。单一的检索方式——无论是纯向量检索还是纯关键词检索——都有各自的盲区。向量检索擅长理解”语义”（知道”好吃的”和”美味的”是一个意思），但可能遗漏精确的关键词匹配；关键词检索擅长精确匹配（绝对不会漏掉”红烧肉”这个词），但无法理解同义词和语义变体。

混合检索将两者结合起来，取长补短，然后再用 RRF（Reciprocal Rank Fusion，倒数排名融合）算法统一排序，给出一个综合的排名结果。

6.2 两种检索器

模块在 setup_retrievers() 中初始化两种检索器：

向量检索器（Vector Retriever）：

基于 FAISS 索引，使用余弦相似度（或内积）计算查询向量与文档向量之间的距离。返回距离最近的 5 个文档。

BM25 检索器（BM25 Retriever）：

BM25 是一种经典的文本检索算法，基于词频（TF）和逆文档频率（IDF）计算相关性。它完全在关键词层面工作，不需要向量化。对于”红烧肉怎么做”这种包含明确关键词的查询，BM25 能非常准确地找到相关文档。

6.3 hybrid_search(query, top_k) 混合检索

这是对外暴露的主检索方法，流程如下：

  1. 分别调用向量检索器和 BM25 检索器的 invoke(query) 方法，各自返回 top-5 的结果。
  2. 将两路结果传入 _rrf_rerank() 进行融合重排。
  3. 返回重排后的前 top_k 个结果（默认 3 个）。

6.4 _rrf_rerank() RRF 重排序算法

RRF（Reciprocal Rank Fusion）是一种简单但有效的多路检索融合算法。它的核心思想是：一个文档在某一检索路径中的排名越靠前，它应该获得越高的分数；如果文档在两条路径中都有出现（说明两种检索方式都认为它相关），它的分数应该累加起来。

算法步骤：

  1. 初始化：创建两个字典——doc_scores 记录每个文档的累计 RRF 分数，doc_object 记录文档对象本身（用于最后重建结果列表）。文档的唯一标识通过 hash(doc.page_content) 生成。

  2. 处理向量检索结果：遍历向量检索返回的文档列表，对排名为 rank 的文档，计算 RRF 分数：

    其中 k 是一个平滑参数（默认为 60），它防止排名非常靠后的文档获得极低的分数，同时也避免排名第一的文档分数过高导致其他结果被完全淹没。rank 从 0 开始（第一名 rank=0，第二名 rank=1…）。

  3. 处理 BM25 检索结果：同样的公式，遍历 BM25 检索结果。如果某个文档在向量检索中已经出现过（相同的 doc_id），RRF 分数会累加到它已有的分数上。这正是融合的关键——出现在两路结果中的文档获得”双倍加分”。

  4. 排序：按累计 RRF 分数从高到低排序。

  5. 构建最终结果：按排序后的顺序重建文档列表，并将最终的 rrf_score 写入每个文档的 metadata 中，方便后续分析。

RRF 算法的一个很好的特性是它天然支持去重——相同的文档内容会生成相同的 hash ID，在字典中自然合并。

6.5 metadata_filtered_search(query, filters, top_k) 带过滤的检索

这个方法在混合检索的基础上增加了元数据过滤能力。例如，用户说”推荐几个简单的素菜”，系统需要先过滤出分类为”素菜”且难度为”简单”的文档，再进行语义检索。

实现逻辑：

  1. 先进行混合检索，但放宽候选范围（top_k * 3），获取更多的候选文档。
  2. 遍历候选文档，逐一检查每个文档的 metadata 是否满足所有过滤条件：
    • 如果过滤条件中的 key 不存在于文档的 metadata 中，直接判定为不匹配。
    • 如果过滤条件的 value 是列表（支持多值匹配），检查文档的 metadata 值是否在列表中。
    • 如果过滤条件是单值，直接比较是否相等。
  3. 收集满足全部条件的文档，数量达到 top_k 后提前终止遍历。
  4. 返回过滤后的文档列表。

七、生成集成模块（rag_modules/generation_integration.py）

7.1 模块职责

生成集成模块是整个 RAG 系统的”出口”，它负责与大语言模型交互，将检索到的菜谱内容组织成用户友好的回答。此外，它还承担了两个智能预处理任务——查询路由和查询重写。

7.2 LLM 初始化

模块使用阿里云百炼平台提供的通义千问大模型，通过 LangChain 的 ChatTongyi 接口调用：

temperature=0.1 是一个较低的值，意味着模型的输出会比较确定和一致，适合需要准确信息的菜谱问答场景。如果设为较高的值（如 0.8），模型的回答会更加”有创意”但也更容易偏离事实。

7.3 _build_context(docs, max_length) 上下文构建

这个方法将检索到的菜谱文档整理成适合输入给 LLM 的上下文字符串：

  1. 如果文档列表为空，返回提示信息”暂无相关食谱信息”。
  2. 遍历文档（从 1 开始编号），为每个文档构建格式化的元数据头信息：
  3. 控制上下文总长度不超过 max_length（默认 2000 字符）。因为 LLM 的上下文窗口有限，加上还要给系统提示词和用户问题留空间，所以需要对内容长度做截断。
  4. 用分隔线 ===== 隔开不同食谱。

7.4 query_router(query) 查询路由

查询路由是一个智能的”分流器”，它先让 LLM 分析用户的问题意图，然后将其分类为三种类型之一：

  1. list（列表查询）：用户想要获取菜品推荐列表，只需要菜名即可。例如：”推荐几个素菜”、”有什么川菜”、”给我3个简单的菜”。对于这种查询，后续只需要调用 generate_list_answer() 给出简洁的菜名列表。

  2. detail（详细查询）：用户想要某个菜的具体制作方法。例如：”宫保鸡丁怎么做”、”红烧肉的步骤”、”蛋炒饭需要什么食材”。对于这种查询，后续会调用 generate_step_by_step_answer() 给出包含食材、步骤、技巧的完整回答。

  3. general（一般查询）：不属于以上两种的其他问题。例如：”什么是川菜”、”炒菜的技巧”、”怎样选肉”。对于这种查询，调用 generate_basic_answer() 给出一般性的回答。

实现方式是通过一个提示词模板让 LLM 进行分类：

安全防线：如果 LLM 没有按要求返回这三个词之一（这在实践中偶尔会发生，比如 LLM 返回了多余的解释文字），代码会强制回退为 general，确保系统不会因为意外的输出而崩溃。流程是：对 LLM 返回的内容做 .strip().lower() 处理后，检查是否在 ['list', 'detail', 'general'] 中，不在则返回 'general'。

这里使用了 LangChain 的链式调用（Chain）模式：

  • RunnablePassthrough() 直接将输入传给下一步。
  • prompt 将输入填充到提示词模板中。
  • self.llm 调用大模型生成回答。
  • StrOutputParser() 将 LLM 的输出对象转换为纯字符串。
  • 管道运算符 | 是 LangChain 的 LCEL（LangChain Expression Language）语法，它创建了一个 RunnableSequence，数据从左到右依次流过每一步。

7.5 query_rewrite(query) 查询重写

查询重写是一个智能的查询优化器。用户输入的问题有时会过于模糊或口语化，直接用于检索效果不好。例如”推荐个菜”这个查询太宽泛了，而重写后的”简单家常菜推荐”则更具体、更容易匹配到相关内容。

LLM 根据以下规则判断是否需要重写：

保持原查询的情况（具体明确）：

  • 包含具体菜品名称：如”宫保鸡丁怎么做”、”红烧肉的制作方法”
  • 明确的制作询问：如”蛋炒饭需要什么食材”
  • 具体的烹饪技巧：如”如何炒菜不粘锅”

需要重写的情况（模糊不清）：

  • 过于宽泛：如”做菜”、”有什么好吃的”
  • 缺乏具体信息：如”川菜”、”素菜”
  • 口语化表达：如”想吃点什么”、”有饮品推荐吗”

重写原则：

  • 保持原意不变
  • 增加相关烹饪术语（如添加”家常菜谱”、”制作方法”等关键词）
  • 优先推荐简单易做的
  • 保持简洁性

这里使用的是 PromptTemplate 而非 ChatPromptTemplate，因为查询重写是一个简单的文本补全任务，不需要多轮对话的结构。

7.6 generate_basic_answer(query, context_docs) 基础回答

用于 general 类型查询的回答生成。使用一个通用的提示词模板，告知 LLM 它是”专业的烹饪助手”，请它根据提供的食谱信息回答用户问题。如果信息不足，要求它诚实说明而不是胡编乱造。

7.7 generate_basic_answer_stream(query, context_docs) 流式基础回答

与 generate_basic_answer 功能相同，但使用流式输出。通过 chain.stream(query) 代替 chain.invoke(query)，结果是一个生成器（generator），每次 yield 一小段文本。这种方式让用户可以在前端看到逐字出现的打字效果，体验更好。

7.8 generate_step_by_step_answer(query, context_docs) 分步骤回答

用于 detail 类型查询。提示词模板包含了详细的回答结构要求：

特别注意制作技巧部分的设计：代码提示词中强调”优先使用原文中的实用技巧”，但”如果原文的附加内容与烹饪无关或为空，可以完全省略此部分”。这是为了避免 LLM 在没有实际技巧时强行填充无关内容（比如把步骤重复一遍当技巧），影响回答质量。

7.9 generate_list_answer(query, context_docs) 列表回答

用于 list 类型查询。不需要调用 LLM，而是直接从文档元数据中提取菜品名称，按数量生成不同格式的回答：

  • 0 个文档：返回”抱歉，没有找到相关的菜品信息。”
  • 1 个文档："为您推荐：红烧肉"
  • 2-3 个文档：带编号的列表，如 "1. 红烧肉\n2. 宫保鸡丁\n3. 清炒白菜"
  • 超过 3 个文档：列出前 3 个，并提示”还有其他 N 道菜品可供选择。”

注意这里做了去重处理——如果多个子块来自同一个父文档，菜名只出现一次。

八、主系统（main.py）

8.1 RecipeRAGSystem 主类

RecipeRAGSystem 是整个系统的”大脑”，它协调四个模块的工作流程。它的核心职责是：

  1. 初始化时加载配置、检查数据路径和 API Key。
  2. 按正确的顺序启动各个模块。
  3. 构建知识库（加载文档 → 分块 → 构建/加载索引 → 初始化检索器）。
  4. 运行问答循环：接收用户问题 → 路由 → 重写 → 检索 → 生成回答。

8.2 init() 初始化

构造函数主要做三件事：

  • 保存配置对象（如果没传就用 DEFAULT_CONFIG）。
  • 检查数据路径 self.config.data_path 是否存在，不存在直接抛 FileNotFoundError。
  • 检查 MOONSHOT_API_KEY 环境变量是否存在（注意：这是项目中的一个已知问题，实际上系统只使用 DASHSCOPE_API_KEY，MOONSHOT_API_KEY 的检查是遗留代码，没有模块真正使用它）。

8.3 initialize_system() 初始化所有模块

按顺序创建四个模块的实例：

  1. DataPreparationModule — 传入数据路径。
  2. IndexConstructionModule — 传入嵌入模型名称和索引保存路径。
  3. GenerationIntegrationModule — 传入 LLM 模型名称、温度和最大 token 数。
  4. RetrievalOptimizationModule 不在这里创建，而是在 build_knowledge_base() 中创建，因为它需要 FAISS vectorstore 和 chunks 作为参数。

8.4 build_knowledge_base() 构建知识库

这个方法是整个系统最复杂的方法之一，它实现了两种路径：

路径一：已有预构建索引

  1. 调用 self.index_module.load_index() 尝试加载磁盘上已保存的 FAISS 索引。
  2. 如果加载成功，虽然向量数据已就绪，但索引本身只保存了向量和文档内容，不包含父文档列表和父子映射关系。因此仍然需要：
    • 重新加载原始文档（self.data_module.load_documents()）
    • 重新分块（self.data_module.chunk_documents()），重建 parent_child_map
  3. 这样既复用了向量计算的结果（节省了 Embedding API 调用的费用和时间），又重建了完整的元数据结构。

路径二：从零构建新索引

  1. 加载原始文档。
  2. 进行文本分块。
  3. 调用 self.index_module.build_vector_index(chunks) 构建新的 FAISS 索引。
  4. 调用 self.index_module.save_index() 保存到磁盘，下次启动就可以走路径一了。

两条路径最终都会：

  • 创建 RetrievalOptimizationModule 实例。
  • 调用 self.data_module.get_statistics() 打印知识库统计信息。

8.5 ask_question(question, stream) 核心问答流程

这是用户交互的核心入口，包含一个精心设计的管道：

第一步：查询路由

让 LLM 判断用户意图是 list、detail 还是 general。

第二步：查询重写

  • 如果路由结果是 list，保持原查询不变（因为列表查询不需要重写，用户说的菜系/类型/难度已经足够）。
  • 如果路由结果是 detail 或 general，调用智能重写优化查询。

第三步：检索

  • 先从用户问题中提取分类和难度的关键词。
  • 如果有过滤条件（比如用户说了”素菜”），使用带过滤的检索。
  • 如果没有过滤条件，使用普通的混合检索。

第四步：获取完整文档

从检索到的子块找回完整的父文档（完整菜谱），供 LLM 生成回答时使用。

第五步：生成回答

  • 如果没有找到任何相关文档，直接返回抱歉信息。
  • list 类型 → generate_list_answer()（不需要 LLM，直接提取菜名）。
  • detail 类型 → generate_step_by_step_answer() 或流式版本。
  • general 类型 → generate_basic_answer() 或流式版本。

8.6 _extract_filters_from_query(query) 关键词过滤提取

这个方法是一个简单但实用的关键词匹配器：

  1. 从 DataPreparationModule.get_supported_categories() 获取所有的中文分类标签（如”荤菜”、”素菜”、”汤品”）。
  2. 检查用户查询中是否包含任意一个分类标签，如果包含就加入过滤条件。
  3. 同样的逻辑处理难度标签（如”简单”、”中等”）。

例如，用户输入”推荐几个简单的素菜”，这个方法会返回 {"category": "素菜", "difficulty": "简单"}（注意：实际代码中只会匹配到一个分类和一个难度即退出循环）。

8.7 search_by_category(category, query) 分类搜索

一个便捷方法，允许用户按分类浏览菜品。内部调用 metadata_filtered_search，指定 filter={"category": category}，返回去重后的菜名列表。

8.8 get_ingredients_list(dish_name) 获取食材

根据菜名搜索相关的文档块，然后让 LLM 生成该菜品的食材清单。调用了混合检索和基础回答生成。

8.9 run_interactive() 交互式问答

启动一个命令行交互循环：

  1. 初始化系统模块。
  2. 构建知识库。
  3. 进入 while True 循环：
    • 读取用户输入。
    • 如果输入为空或退出命令（”退出”/“quit”/“exit”），结束循环。
    • 询问用户是否使用流式输出。
    • 调用 ask_question() 获取回答。
    • 流式模式下逐块打印（print(chunk, end="", flush=True)），flush=True 确保每个字符块立即显示。
    • 非流式模式下一次性打印完整回答。
  4. 处理 KeyboardInterrupt（Ctrl+C）优雅退出。

九、RAG 系统的数据流全貌

让我们用”用户问’红烧肉怎么做’”这个例子，完整走一遍数据流：

步骤 1：系统启动

  1. RecipeRAGSystem.__init__() 加载配置，检查路径和 API Key。
  2. initialize_system() 创建四个模块实例。
  3. build_knowledge_base() 加载 323 个菜谱 Markdown 文件 → 按标题分割成若干子块 → 向量化 → 构建 FAISS 索引 → 初始化 BM25 检索器。

步骤 2：接收查询

用户输入："红烧肉怎么做"

步骤 3：查询路由

query_router("红烧肉怎么做") → LLM 分析 → 返回 "detail"

步骤 4：查询重写

query_rewrite("红烧肉怎么做") → LLM 判断这是具体查询 → 保持原样返回 "红烧肉怎么做"

步骤 5：关键词过滤提取

_extract_filters_from_query("红烧肉怎么做") → 不包含分类或难度关键词 → 返回空字典 {}

步骤 6：混合检索

hybrid_search("红烧肉怎么做", top_k=3) →

  • 向量检索器返回 5 个语义最相关的子块
  • BM25 检索器返回 5 个关键词最匹配的子块
  • RRF 算法融合两路结果，取前 3 个

步骤 7：父子文档映射

get_parent_documents(检索到的3个子块) → 从 parent_child_map 找回对应的完整父文档 → 得到完整版《红烧肉》菜谱

步骤 8：生成回答

因为路由类型是 detail，调用 generate_step_by_step_answer("红烧肉怎么做", [红烧肉完整文档]) →

  • 构建包含完整菜谱内容的上下文字符串
  • 填入提示词模板（包含菜品介绍、食材、步骤、技巧的结构要求）
  • 调用通义千问 LLM 生成结构化回答
  • 返回给用户

步骤 9：输出

用户看到包含 🥘 菜品介绍、🛒 所需食材、👨‍🍳 制作步骤、💡 制作技巧的完整回答。

十、关键技术决策与设计理念

10.1 为什么使用父子文档模式？

在 RAG 系统中，文档粒度需要在”检索精准度”和”上下文完整性”之间做权衡。如果直接用完整文档做检索，文档太长，语义被稀释，检索不准；如果只用小块做检索和生成，LLM 只能看到片段，生成质量差。父子文档模式同时解决了这两个问题——小块用于精准检索，大块用于完整生成。

10.2 为什么使用混合检索？

纯向量检索虽然在多数情况下表现很好，但有两个弱点：

  • 对专有名词（如”宫保鸡丁”）的精确匹配不如关键词检索。
  • 嵌入模型可能对某些领域的术语理解不够好。

BM25 正好弥补了这些弱点。而 RRF 算法提供了一个不需要调参的融合方案——不需要手动设置向量检索和关键词检索的权重，算法自动根据排名来融合。

10.3 为什么需要查询路由？

不同类型的查询需要不同风格的回答。如果一个用户说”推荐几个素菜”，你给他返回”素菜分几步怎么做”的长篇大论显然不合适；反之如果用户问”红烧肉怎么做”，只返回一个菜名列表也不够。查询路由让系统能根据用户意图自适应地调整回答策略。

10.4 为什么 temperature 设得这么低？

菜谱问答是一个需要准确性的任务。用户需要精确的食材用量和正确的步骤顺序。temperature=0.1 使 LLM 的输出更确定、更可预测、更不容易”自由发挥”出错误的菜谱。
