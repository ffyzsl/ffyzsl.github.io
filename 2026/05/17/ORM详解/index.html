简单说，数据库就是一个有组织的数据存储文件。你电脑上的 .db 文件就是数据库。

1.2 没有 ORM 时，怎么和数据库打交道？

你直接写 SQL 语句（Structured Query Language，结构化查询语言），然后通过数据库驱动发给数据库。就像这样：

1.3 这种方式有什么痛点？

  1. 写 SQL 字符串很容易出错：表名拼错、字段名写漏，IDE 帮不了你，要到运行时才发现。
  2. SQL 和 Python 代码混在一起：代码很乱，读起来费劲。
  3. 查询结果是元组/字典：row[0] 这种写法完全看不出是 id 还是 name，写错了也不报错。
  4. 不同数据库的 SQL 语法略有不同：换数据库（比如从 SQLite 换到 PostgreSQL）要修改 SQL 语句。
  5. 没有类型检查：你插入的数据类型对不对，要到执行时才知道。

ORM 就是来解决这些问题的。

2. ORM 是什么？

2.1 一句话解释

ORM（Object-Relational Mapping，对象关系映射）是一种技术：让你用操作 Python 对象的方式来操作数据库，不用手写 SQL。

2.2 用人话打个比方

不用 ORM 时，你和数据库交流是这样的：

ORM 就是你和数据库之间的翻译官：你把 Python 对象交给它，它自动翻译成 SQL 发给数据库；数据库返回原始数据，它又自动翻译回 Python 对象给你。

2.3 ORM 的核心映射关系

ORM 做的最核心的事情就是映射（Mapping）：

数据库中的概念 映射到 → Python 中的概念
表（Table） → 类（Class）
表中的一行（Row） → 类的一个实例（Instance）
列（Column） → 实例的属性（Attribute）
表之间的关系 → 对象之间的引用

2.4 ORM 的优缺点

优点：

  • 写 Python 代码比写 SQL 字符串舒服，IDE 有自动补全和类型检查
  • 换数据库时（SQLite → PostgreSQL）几乎不用改代码
  • 自动防 SQL 注入（参数化查询）
  • 代码更整洁，逻辑更清晰

缺点：

  • 有学习成本
  • 复杂查询的性能不如手写 SQL
  • 对数据库的精细控制不如原生 SQL

对于本项目这种桌面聊天应用来说，查询都很简单，ORM 的缺点完全不受影响，优点则是巨大的。

3. SQLAlchemy 是什么？

3.1 一句话解释

SQLAlchemy 是 Python 生态中最流行的 ORM 库。它同时支持同步和异步操作，支持几乎所有数据库（SQLite、PostgreSQL、MySQL 等）。

你可以把它理解为一个加强版的翻译官：它不仅能翻译 Python ↔ SQL，还有一大堆附加功能（连接池、会话管理、迁移工具等）。

4.4 Session（会话）—— 和数据库的”一次通话”

是什么？

SQLAlchemy 中的 Session 代表和数据库的一次”对话”。 在这个对话中，你可以做增删改查，所有操作会先暂存，最后一次性提交。

⚠️ 注意区分两个”Session”：

  • 本项目的 Session（大写 S）= ORM 模型，对应 sessions 表
  • SQLAlchemy 的 session（小写 s）= 数据库会话，用于执行数据库操作

这两个概念刚好同名，但完全不同。后面我会用”ORM Session”和”DB session”来区分。

DB session 的工作模式

这种”暂存再提交”的模式叫做事务（Transaction）：

  • 要么全部成功（commit）
  • 要么全部撤销（rollback）

保证了数据一致性，不会出现”改了一半”的情况。

expire_on_commit=False 是什么意思？

默认情况下，commit() 之后 SQLAlchemy 会让所有已加载的对象”过期”——下次访问属性时会重新查数据库。设 False 后对象不会过期，你 commit 完还能用。

FastAPI 中的使用：依赖注入

FastAPI 的依赖注入机制：每个 HTTP 请求到来时，FastAPI 自动调用 get_db()，生成一个新的 DB session 注入到你的处理函数中。请求结束后 session 自动关闭。

关于 Depends(get_db) 的更多细节：

  • Depends 是 FastAPI 的机制，告诉框架”这个参数的值请调用 get_db 函数来获取”
  • get_db 是一个异步生成器（async def + yield），yield 前面的代码在请求开始时执行，后面的代码在请求结束时执行
  • 这样就实现了”一个请求一个 DB session，用完自动关闭”

4.5 Metadata（元数据）—— 所有表的”花名册”

是什么？

Metadata 是一个注册表，记录了你定义的所有表的信息。 它知道你的数据库里有什么表，每个表有什么列，列是什么类型。当你定义模型类并继承 Base 时，SQLAlchemy 自动把表信息注册到 Base.metadata 里。

create_all 做了什么事？

  1. 读取 Base.metadata 中注册的所有表
  2. 检查数据库，哪些表还不存在
  3. 对于不存在的表，生成 CREATE TABLE SQL 并执行
  4. 对于已存在的表，不做任何事（不会删表或改表）

run_sync 是什么？

因为 Base.metadata.create_all 是同步函数，而我们在异步环境（FastAPI）中。run_sync 的作用就是：在异步代码中安全地执行同步函数，它会在另一个线程中执行同步函数，不阻塞事件循环。

5. aiosqlite 是什么？

5.1 从 sqlite3 到 aiosqlite

Python 标准库自带 sqlite3 模块，可以直接操作 SQLite 数据库。但它是同步的——执行 SQL 时会阻塞当前线程，直到数据库返回结果。

在异步框架（FastAPI）中，阻塞线程会影响性能：当一个请求在等数据库时，线程被卡住，其他请求只能排队等着。

aiosqlite 是 sqlite3 的异步版本。 它让你可以用 await 来执行数据库操作：

5.2 aiosqlite 在本项目中的角色

在本项目中，你不直接使用 aiosqlite。它的角色是”幕后工作者”——SQLAlchemy 通过 aiosqlite 来和 SQLite 数据库通信：

DATABASE_URL 中的 +aiosqlite 就是告诉 SQLAlchemy：用 aiosqlite 作为底层驱动。

5.3 为什么选 SQLite + aiosqlite？

原因 说明
零配置 SQLite 不需要安装数据库服务，就是一个文件
适合桌面应用 本项目的数据库在用户本地，不共享，SQLite 最合适
异步支持 aiosqlite 让 SQLite 也能用于异步框架
轻量 不需要 MySQL/PostgreSQL 那样的独立进程

6.3 run_sync() —— 异步世界里的同步”安全区”

有些 SQLAlchemy 操作是同步的不支持异步，比如 Base.metadata.create_all()。于是 SQLAlchemy 提供了 run_sync()：

它的工作原理：

  1. 在主线程之外开一个辅助线程
  2. 在辅助线程里执行 Base.metadata.create_all（同步函数）
  3. 主线程用 await 等待辅助线程完成
  4. 这样主线程（事件循环）不会被同步代码阻塞

Update（更新）

关键点：不需要显式调用 update()。你只需要修改 Python 对象的属性，SQLAlchemy 会自动跟踪哪些属性被修改了，commit() 时生成相应的 UPDATE SQL。

但是前面 add_message 中的 session.updated_at 需要手动更新，这是因为虽然 SQLAlchemy 能跟踪 session.name = "xxx" 这种直接赋值，但它不会自动意识到”关联了新的 Message”也意味着 Session 应该被标记为修改。这是一个值得注意的边界情况。

Delete（删除）

由于配置了 cascade="all, delete-orphan"，删 Session 时，该 Session 的所有 Message 也自动被删。

这是一种最小化迁移策略——没有版本号追踪，没有回滚支持，只做”加列”一件事。对于桌面应用，这足够了。

9. 常见问题 FAQ

Q1: Mapped[T] 和普通类型注解有什么区别？

Mapped[T] 是 SQLAlchemy 2.0 的核心类型。它同时做两件事：给 IDE 提供类型信息，给 SQLAlchemy 提供列映射信息。

Q2: session = await db.get(Session, id) 和 select(Session).where(...) 什么时候用哪个？

  • db.get(Model, primary_key_value) —— 按主键查一条，最简单直接
  • select(Model).where(...) —— 任何其他查询（按非主键查、条件查询、排序、分页）

db.get() 内部等价于 select(Model).where(Model.id == pk)，但它更简洁，而且会先查一级缓存。

Q3: 为什么 session_id 用 String 而不是 UUID 类型？

SQLAlchemy 支持 UUID 类型，但 SQLite 不原生支持 UUID。用 String 直接存 UUID 的字符串表示更简单，避免类型转换的坑。

Q4: ForeignKey("sessions.id") 引用的是表名还是类名？

是数据库表名（sessions，不是 Python 类名 Session）。ForeignKey 括号里写的是 表名.列名。

Q5: 为什么不在 Message 模型里 from backend.models.session import Session？

如果你这样做，就会产生循环导入：

  • session.py 需要 Message（定义 messages relationship）
  • message.py 需要 Session（定义 session relationship）

本项目的解决方法是使用 TYPE_CHECKING：

在运行时，TYPE_CHECKING 是 False，所以这个 import 不会执行——但类型检查器（mypy / Pyright）会把它当真，IDE 也能正常补全。

Q6: expire_on_commit=False 到底有什么用？

假设你这样做：

  • 默认（expire_on_commit=True）：commit 后对象过期，print(session.name) 会触发一次隐式数据库查询来刷新数据。如果此时 DB session 已关闭，直接报错。
  • expire_on_commit=False：commit 后对象不过期，直接返回已有值，不会触发隐式查询。

对于 FastAPI 这种”请求结束就关 session”的模式，False 更合适。

Q7: aiosqlite 和 sqlite3 到底什么关系？

aiosqlite 不是重写 SQLite，而是把 sqlite3 的调用放到后台线程中执行，用 asyncio 的事件循环机制来实现”不阻塞”的效果。

Q8: 为什么要有两个引擎（async engine 和 sync engine）？

因为 SQLAlchemy 的某些操作只支持同步：

  • Base.metadata.create_all() —— 建表
  • inspect() —— 检查数据库结构（看有哪些表、哪些列）

在 init_db() 中，这些同步操作通过 conn.run_sync() 执行——它需要一个同步引擎来创建同步连接。

日常 CRUD 操作用异步引擎，建表迁移用同步引擎。

Q9: await conn.run_sync(some_function) 中的 conn 是异步连接还是同步连接？

是异步连接（来自 engine.begin()，async engine）。run_sync() 方法在异步连接对象上被调用，它会在内部使用同步引擎的线程池来执行传入的同步函数。
