1. 什么是 ReAct？

1.1 名字的由来

ReAct = Reasoning（推理） + Acting（行动）

这是一种让大语言模型（LLM）既能”想”又能”做”的模式。传统的聊天机器人只能根据训练数据回答问题，而 ReAct Agent 可以调用外部工具来获取实时信息，并基于这些信息进行多步推理。

1.3 ReAct 循环的三个角色

角色 谁来写 作用
Thought（思考） 模型 分析当前情况，决定下一步做什么
Action（行动） 模型 调用一个工具（查数据库、读文件、做计算）
Observation（观察） 系统 工具的实际返回结果，反馈给模型
Answer（回答） 模型 当信息足够时，给出最终答案

关键理解：Thought 和 Action 由模型输出，Observation 由系统（你的代码）提供。这三者交替循环，直到模型认为信息足够了，输出 Answer 结束。

2. 项目概述

2.1 这个项目做什么？

一个电商客服 AI Agent，用户可以：

  • 询问商品信息（价格、库存、品牌等）
  • 查询优惠策略
  • 计算最终成交价格
  • 进行简单的寒暄对话

决策 说明 为什么这样设计

4. 阶段 1：环境准备与配置系统

4.1 目标

搭建项目骨架，创建配置文件，理解什么信息该放哪里。

4.2 为什么需要三个配置文件？

文件 存什么 进 git 吗 例子
config.json 非敏感的运行参数 ✅ 进 模型名、temperature、max_iterations
.env 敏感的密钥 ❌ 不进 DASHSCOPE_API_KEY=sk-xxx
.env.example 密钥的模板 ✅ 进 DASHSCOPE_API_KEY=your-key-here

核心原则：代码和配置分离，敏感信息和普通配置分离。

想象你把项目分享给同事：

  • 同事看到 .env.example 知道需要申请什么 Key
  • 同事创建自己的 .env 填入真实 Key
  • .gitignore 确保 .env 不会被提交到 GitHub

4.3 config.json 详解

temperature 参数说明：

  • temperature = 0 → 模型每次输出几乎相同，适合需要稳定输出的场景
  • temperature = 1.0 → 输出有一定随机性，适合客服对话
  • temperature = 2.0 → 输出非常随机，可能出现不稳定的结果

4.4 load_config() 函数

json.load(f) 读取 JSON 文件内容并转换为 Python 字典。非常简单，但在整个项目中会被多处使用。

4.5 python-dotenv 的工作机制

load_dotenv() 扫描项目根目录的 .env 文件，把每一行 KEY=VALUE 注入到系统环境变量 os.environ 中。之后你就可以用 os.environ.get("KEY_NAME") 读取。

5. 阶段 2：工具层 —— Act 的部分

5.1 什么是 Tool（工具）？

在 ReAct 模式中，Tool = Agent 的手。Agent 不是全知全能的，它通过工具获取外部信息：

  • 商品有哪些？→ 工具 1 查数据库
  • 有什么优惠？→ 工具 2 读优惠文件
  • 最终价格多少？→ 工具 3 做计算

每个工具都是一个普通的 Python 函数，接收字符串参数，返回结果。

SQLite 是一个嵌入式数据库，数据存在一个 .db 文件中，不需要安装数据库服务器。Python 标准库自带 sqlite3 模块，开箱即用。

5.4 工具 3：calc.py — 安全计算

5.4.1 为什么不能用裸 eval()？

eval() 会执行任意 Python 代码。在大模型时代，攻击者可以通过 prompt injection 诱导模型输出恶意表达式。**凡是接受外部输入的代码，绝不能用裸 eval()**。

5.4.2 安全方案：正则白名单 + 受限 eval

两道防线的原理：

防线 作用 阻挡了什么
正则 ^[\d\s+\-*/().]+$ 只放行数学符号 __import__、os.system 中包含字母和点号，无法通过
{"__builtins__": {}} 禁用所有内置函数 即使有办法绕过正则，也无法调用任何 Python 内置功能

为什么 _ 很重要：正则中不包含 _（下划线），所以 __import__、__class__、__bases__ 等 Python 特殊属性全部无法拼写。这是关键的安全保障。

这是最基础的 LLM API 概念，理解它对后续学习至关重要：

7.1 CustomerServiceAgent 类

__call__ 是 Python 的魔术方法（dunder method），让类的实例可以像函数一样被调用。这让代码更简洁自然。

7.3 System Prompt 设计

System Prompt 是 Agent 的”行为准则”，它告诉模型：你是谁、能做什么、怎么回答、用什么格式。

7.3.1 一个有效的 System Prompt 包含什么

7.3.2 本项目 System Prompt 逐段解析

第一段：角色定义

告诉模型”你是谁”——一个电商客服。这设定了模型的语调和知识范围。

第二段：核心规则

这是最重要的部分。如果不加这些规则，模型会一次性输出整个 Thought→Action→Observation→Answer 链（包括它自己编造的假 Observation），导致整个 ReAct 循环失效。详见 10.2 模型幻觉问题。

第三段：工具列表

工具名必须和 main.py 中 tools 字典的 key 完全一致，否则会找不到工具。

第四段：输出格式

精确的格式定义，确保正则表达式能正确解析。tool_name 和 argument 之间用 : （冒号+空格）分隔。

第五段：Few-shot 示例

示例用 --- 分隔每个轮次，明确告诉模型”每次只输出一段”。这是 Few-shot learning 的核心——通过例子而不是规则来教学。

为什么要用英文 Prompt？ 百炼模型（qwen-plus）在英文指令上的遵循度通常优于中文指令。英文 Prompt 结构更清晰，模型更容易精确理解格式要求。但最终回答要求是中文。

7.4 双后端分支处理

两个路径的核心区别：

  • Ollama：chat_completions_create() 返回 str（模型的回复文本）
  • 百炼：client.chat.completions.create() 返回一个对象，文本在 completion.choices[0].message.content 中

8.1 为什么是两层循环？

8.4 Observation 回传机制

这是 ReAct 文本式工具调用最关键的设计。

看消息列表的演变过程：

关键设计：Observation 以 user 角色注入。这是因为模型对 user 消息的重视程度最高——user 代表”真实的外部信息”，assistant 代表”自己说过的话”。将工具结果以 user 角色注入，模型会更认真地处理这些信息。

8.5 get_client() — 后端选择逻辑

.get('use_model', True) 的含义：读取 use_model 的值，如果这个 key 不存在则默认为 True。这是一种防御性编程。

8.6 max_iterations — 安全阀

为什么需要这个限制？

  • 如果模型一直输出 Action 而不输出 Answer → 死循环
  • 如果模型格式不规范，正则解析失败 → 无限循环
  • max_iterations 是最后的安全阀，避免程序卡死

10. 常见问题与解决方案

10.1 “你好”导致死循环

现象：输入”你好”，程序达到 max_iterations 仍未结束。

原因：模型输出 你好！有什么可以帮助你的？，但没加 Answer: 前缀。正则匹配不到 Action，也匹配不到 Answer，走 fallback 继续循环。

解决：添加第三个分支——如果模型输出了内容但不符合任何格式，直接当作答案处理：

10.2 模型幻觉：自己编造 Observation

现象：模型一次输出完整的 Thought→Action→Observation→Answer，其中 Observation 是编造的。

示例：

根因：

  1. System Prompt 中的示例展示了完整的 Thought→Action→Observation→Answer 链，模型学会了”一次性输出全部”
  2. 工具参数解析用字符串切割而非正则 groups，导致参数中混入了后续文本

解决：

  1. System Prompt 加 CRITICAL RULES：每次只输出一个 Action，绝不自己编 Observation
  2. 用正则 groups 精确提取第一行的工具名和参数
  3. 示例用 --- 分隔每个轮次

10.3 API Key 未设置错误

现象：OpenAIError: The api_key client option must be set

原因：没有调用 load_dotenv()，导致 .env 文件未被读取。

解决：在使用 os.environ.get() 之前调用 load_dotenv()。

10.4 工具返回空结果

可能原因：

  1. 工具参数包含了多余的空格或换行符 → 在调用前 strip() 参数
  2. 数据库/文件路径不对 → 确保 CWD 是项目根目录
  3. 文件编码问题 → 明确指定 encoding='utf-8'
