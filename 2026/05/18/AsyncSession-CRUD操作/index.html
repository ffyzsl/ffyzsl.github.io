AsyncSession CRUD操作

发表于 2026-05-18

1. 为什么需要 AsyncSession？

在 FastAPI 这类异步 Web 框架中，如果仍然使用同步的 SQLAlchemy Session，数据库 IO 会阻塞整个事件循环，异步带来的并发优势荡然无存。SQLAlchemy 1.4+ 引入了 AsyncSession，让我们可以用 async/await 的方式操作数据库，和 FastAPI 的异步生态无缝衔接。

同步 vs 异步对比

核心思路：凡是涉及网络 IO 的操作（execute、commit、refresh）都需要 await；纯内存操作（add、修改属性）则不需要。

2. 环境搭建

2.1 异步引擎与会话工厂

expire_on_commit=False 为什么重要？

默认情况下，SQLAlchemy 在 commit() 后会使所有已加载对象过期（expire），下次访问属性时会自动发出新的 SELECT。但在异步环境下，这个”自动”会尝试在非 async 上下文中执行 IO，直接抛出 MissingGreenlet 错误。关掉它，提交后对象依然可用。

2.2 FastAPI 依赖注入

async with 保证了请求结束时 session 被正确关闭，连接归还池中。

3. 模型定义（SQLAlchemy 2.0 风格）

Mapped + mapped_column 是 2.0 新式写法，类型提示更清晰。

4. Create（新增）

4.1 单条插入

注意：add() 不需要 await，但 commit() 必须 await

add() 只是把对象注册到 session 的待处理列表中，不涉及任何 IO。真正发 INSERT 语句是在 commit()（或 flush()）时，所以只有后者需要 await。

4.2 批量插入

方式 适用场景
add_all 数据量小、需要 ORM 事件/级联
insert().values() 批量 数据量大、追求性能、不关心 ORM 特性

5.4 统计查询

scalar() vs scalars() vs scalar_one()

方法 返回值 用途
scalars().all() list[Model] 多行结果
scalars().first() Model | None 取第一行
scalar() 单个值 聚合查询 count/sum/avg
scalar_one() 单个 Model 确认只有一行，多了抛异常

6. Update（更新）

6.1 先查后改（推荐）

ORM 的 脏检查（Dirty Check） 会自动对比对象当前值和加载时的值，只生成变化的字段的 UPDATE 语句。

6.2 批量更新（不查直接改）

先查后改适合单条、需要业务逻辑判断的场景；批量更新适合按条件批量修改，一次 SQL 搞定。

7. Delete（删除）

7.1 单条删除

delete() 需要 await！

delete() 和 add() 不同，它会立即发出 SQL（标记对象为 deleted 状态并 pre-flush），所以必须 await。

8. 关联查询与预加载

MissingGreenlet —— 异步开发的头号陷阱

规则：在异步环境下，所有关联属性必须显式预加载。 selectinload 用一条额外的 SELECT IN 加载关联数据，性能好且代码清晰；joinedload 用 JOIN 一次查出，适合一对一或多对一。

9. 事务管理

with_for_update=True 加行级锁，防止并发扣款时出现超卖。FastAPI 依赖注入的 async with 会在异常时自动 rollback。
