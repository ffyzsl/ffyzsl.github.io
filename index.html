1. List (双向链表)

在Go语言中，标准库 container/list 提供了一个双向链表的实现。

核心认知（面试重点）

  • 出场率极低：在Go的实际开发中，我们极少使用 list。99%的场景下，我们都会使用切片（Slice）。
  • 为什么不用List？ 因为切片的底层是连续内存，对 CPU缓存（CPU Cache）极其友好，遍历速度碾压链表。而链表的节点在内存中是分散的，不仅遍历慢，还会给GC（垃圾回收）带来额外的扫描压力。
  • 什么时候用？ 只有在需要极其频繁地在中间插入/删除元素，且数据量极大的特定算法场景（如LRU缓存淘汰算法）下，才会考虑使用。

2. nil (空值) —— 面试连环坑

在Java中叫 null，在C中叫 NULL，在Go中叫 nil。但Go的 nil 有着极其特殊的底层逻辑。

哪些类型可以是 nil？

记住，只有指针、切片、Map、通道（Channel）、函数、接口（Interface）这6种引用/指针类型的零值才是 nil。
基础类型（int, string, bool）和结构体（struct）永远不能是 nil。

致命面试题：nil 是关键字吗？

答：不是。 nil 只是Go语言预定义的一个标识符。理论上你可以定义一个变量叫 nil（虽然绝对不要这么做），这在Java/C++中是不可能的。

终极陷阱：带有类型的 nil 接口 (Typed Nil)

这是Go面试中最经典的坑，没有之一。一个接口为 nil，必须是它的“类型”和“值”都为 nil。

3. make 和 new 的区别与原理

这是Go语言内存分配的两个核心内置函数。面试官只要问基础，必问这道题。

核心总结表（建议截图/默写）

特性 new(T) make(T, args)
适用类型 所有类型（如 int, struct 等） 仅限 Slice, Map, Channel
返回值类型 返回指向该类型的指针 *T 返回该类型的值（引用本身） T
内存操作 申请内存，并将内存清零 申请内存，并初始化底层数据结构

深入原理解析

1. new 的原理

new(T) 的作用很简单：找一块内存，把里面填满 0（零值），然后把这块内存的地址给你。
它不会做任何复杂的初始化工作。

2. make 的原理

为什么 Slice、Map、Channel 必须用 make？
因为这三个家伙在Go底层是复杂的结构体。仅仅把内存清零是不能用的，必须初始化它们的内部字段（比如切片的长度和容量、Map的哈希桶指针、Channel的环形队列和锁）。

反面教材：如果用 new 去创建 Map 会怎样？（面试常问）

面试标准话术（背诵版）

面试官：请说一下 Go 语言中 make 和 new 的区别？

你：它们都是用来分配内存的内置函数，但有三个核心区别：

  1. 服务对象不同：make 专属用于切片、Map和Channel这三种引用类型的初始化；new 可以用于任何类型（通常是结构体或基本类型）。
  2. 返回值不同：make 返回的是这三种类型的实例值本身；而 new 返回的是指向分配内存的指针。
  3. 工作机制不同：new 只是在内存中申请一块空间并置为零值；而 make 除了申请内存，还会对这三种复杂类型的底层数据结构进行初始化（例如分配Map的哈希桶、设置切片的len和cap等）。如果用 new 来分配 Map，得到的是一个指向 nil map 的指针，直接写入会引发 panic。

1. Range

range 是Go语言中用于遍历数组、切片、Map、字符串和通道（Channel）的关键字。看似简单，但这里藏着Go语言最经典的面试坑。

核心机制

  • 值拷贝：for i, v := range slice 中，这个 v 是切片元素的副本。
  • 地址复用（Go 1.22之前）：在Go 1.22版本之前，v 的内存地址在整个循环中是不变的。虽然Go 1.22修复了闭包捕获的问题，但值拷贝的特性依然存在。

3. Map (映射/哈希表)

Map是Go语言中最强大的键值对结构。它是引用类型。

核心规则（背下来！）

  1. 无序性：Map的遍历顺序是随机的！不要依赖遍历顺序。
  2. Key的限制：Key必须是可比较的类型（如int, string, bool, 指针, 结构体）。切片、函数、Map不能做Key。
  3. 零值陷阱：声明但未初始化的Map是 nil。读取 nil map 返回零值（安全），但向 nil map 写入会导致 panic（崩溃）。

5. Map 的删除和清空

删除单个键值对

使用内置函数 delete(map, key)。

  • 如果 key 存在，删除之。
  • 如果 key 不存在，什么都不发生，也不会报错。

清空 Map

  • Go 1.21 之前：没有直接清空的函数。通常做法是 m = make(map[...]) 创建一个新的，旧的交给GC回收；或者遍历删除（效率低）。
  • Go 1.21 之后：新增了内置函数 clear(m)，效率极高。

6. Map 的多键索引 (Multi-key)

Go不支持 map[k1, k2]val 这种语法。如果你需要用两个ID来确定一个值，怎么办？

方案一：嵌套 Map (不推荐)

map[string]map[string]int。初始化麻烦，每一层都要检查 nil。

方案二：结构体作为 Key (推荐)

这是Go语言的一个妙招。因为结构体只要字段可比较，它本身就是可比较的。

7. sync.Map (并发安全 Map)

这是面试的高频考点。

为什么需要它？

标准的 map 不是并发安全的！
如果在多个 goroutine 中，一个在读，一个在写（或者两个都在写），程序会直接 Panic 崩溃（fatal error: concurrent map writes）。

解决方案

  1. 加锁：使用 sync.RWMutex 保护标准 map（最常用）。
  2. sync.Map：Go语言自带的并发安全 map。

sync.Map 特点

  • 开箱即用：不用 make，直接 var m sync.Map。
  • 性能场景：适合读多写少的场景。如果写操作非常频繁，性能不如“标准Map + 锁”。
  • 操作方式：不能用 m[k]，必须用特定方法。

  • 普通 Map 读写不安全，并发写必死。

  • Map 遍历顺序是随机的。

1. 数组 (Array)

在Go中，数组是值类型（Value Type），这一点至关重要。

核心概念

  • 长度固定：声明时长度是多少，就是多少，不能动态扩容。
  • 类型包含长度：[3]int 和 [4]int 是完全不同的两种数据类型，不能互相赋值。
  • 值传递：当你把一个数组传递给函数时，Go会拷贝整个数组。如果数组很大，这会非常消耗内存。

面试官爱问：

A: 因为数组是值传递，传参时发生了全量拷贝。应该改用切片（Slice）或者传递数组指针。

3. 切片 (Slice) —— 重中之重

切片是Go中最常用的数据结构。它本质上是对数组的一个引用（Reference）。

核心结构（背下来！）

切片在运行时是一个结构体，包含三个字段：

  1. Pointer: 指向底层数组中切片开始位置的指针。
  2. Length (len): 切片当前包含的元素个数。
  3. Capacity (cap): 从切片开始位置到底层数组末尾的元素个数。

代码演示

面试官爱问：
Q: nil 切片和空切片（Empty Slice）有什么区别？
A:

  • var s []int 是 nil 切片（底层指针为 nil）。
  • s := []int{} 或 make([]int, 0) 是空切片（底层指针不为 nil，指向一个zerobase地址）。
  • 但在JSON序列化时，nil切片变成 null，空切片变成 []。

4. append() 与 扩容机制

append 用于向切片追加元素。当底层数组容量不够时，Go会创建一个新的、更大的数组，并将旧数据拷贝过去。

扩容策略（面试必问）

  • Go 1.18之前：容量 < 1024 时，每次翻倍；>= 1024 时，每次增加 25%。
  • Go 1.18之后：算法变得更平滑，不再是严格的翻倍，但大致逻辑依然是：小切片翻倍，大切片渐进式增长。

5. 切片复制 (Copy)

由于切片是引用，简单的 s1 = s2 只是拷贝了指针、长度和容量，它们依然共享底层数组。如果你想彻底分离，必须用 copy。

核心规则

copy(dst, src) 只会复制 min(len(dst), len(src)) 个元素。也就是说，目标切片必须先分配好足够的长度。

总结：面试避坑指南

  1. 数组是值，切片是引用：这是万恶之源，一定要分清。
  2. append 必须赋值：s = append(s, x)，不要只写 append(s, x)。
  3. 切片截取不拷贝：subSlice := s[1:3]，修改 subSlice 会影响 s。如果不想影响，请用 copy。
  4. 大数组的小切片：如果你有一个很大的数组，但只切片引用了其中一小段，整个大数组都会被留在内存中无法释放。解决方法是 copy 那一小段出来，让大数组去死（被GC回收）。

1. 变量逃逸分析 (Escape Analysis)

什么是逃逸分析？

在C/C++中，开发者需要手动管理内存（分配和释放）。而在Go中，编译器非常聪明，它会自动决定一个变量是分配在栈（Stack）上，还是分配在堆（Heap）上。这个决定的过程，就叫逃逸分析。

  • **栈 (Stack)**：分配快，释放快，函数结束自动回收。
  • **堆 (Heap)**：分配慢，需要垃圾回收器（GC）来回收，会带来额外的CPU消耗。

开放-封闭原则，是说软件实体（类，模块，函数等等）应该可以拓展，但是不可修改。对于扩展是开放的，对于修改是封闭的。

在设计一个模块时，应当使得这个模块可以在不被修改的前提下被扩展。
